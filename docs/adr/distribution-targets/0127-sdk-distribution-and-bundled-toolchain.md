# ADR-0127: SDK distribution and bundled toolchain

## Status

Accepted.
Amends ADR-0080 (SDK platform matrix — see §Platform matrix below).
Amends ADR-0121 (libc++ is a modified fork, not stock LLVM libc++).

## Context

Cart authors need a cross-compilation toolchain targeting RV32IMAFC ILP32
to build cart binaries. Without a bundled toolchain, setup varies per
platform: on macOS and Linux a RISC-V cross-compiler can be installed via
package managers, but packages differ in version and configuration across
distributions; on Windows no mainstream package manager ships one at all.

ADR-0080 established the supported development platforms: macOS ARM64,
Windows x86-64, Linux x86-64, Linux ARM64. ADR-0121 established that the
SDK ships `libc++.a` and `libc++abi.a` for C++ carts and that
`std::random_device` terminates within the cart sandbox. It did not specify
how the libc++ artifact is sourced, built, or what modifications it carries.

## Decision

### Platform matrix

The SDK is distributed as a pre-packaged download for four host platforms.
This amends ADR-0080's toolchain discussion: rather than requiring cart
authors to install a cross-compiler separately, the SDK bundles one.

| Download | Host platform |
|---|---|
| `blyt-sdk-{version}-windows-x64.zip` | Windows x86-64 |
| `blyt-sdk-{version}-linux-x64.tar.gz` | Linux x86-64 |
| `blyt-sdk-{version}-linux-arm64.tar.gz` | Linux ARM64 |
| `blyt-sdk-{version}-macos-universal.tar.gz` | macOS Universal (ARM64 + x86-64) |

The macOS package is a universal binary, covering both Apple Silicon and
Intel Macs in a single download. LLVM's official releases already ship as
universal macOS binaries. `blyt` requires two Cargo target builds
(`aarch64-apple-darwin` + `x86_64-apple-darwin`) combined with lipo.
This amends ADR-0080, which previously excluded Intel Mac; the change is
a packaging decision with no architectural consequence.

### Bundled LLVM Clang and LLD

The SDK bundles pre-built Clang and LLD from official LLVM project releases.
LLVM is not built from source as part of the SDK build pipeline. A single
Clang binary cross-compiles to any target, including `riscv32-unknown-none-elf`
for RV32IMAFC ILP32 carts, without a separate per-target GCC cross-compiler
build.

The bundled Clang and LLD are the only supported compilers for cart C and
C++ code in this SDK. Cart authors do not need to install a separate
compiler.

**Emscripten** is not bundled. Its `emsdk` tool provides a self-contained
installation mechanism for the WASM cart build target; it is documented
separately. Emscripten's update cadence and download size make embedding
it in the cart SDK impractical.

### Two-tier artifact structure

SDK contents split into two tiers:

**Per-host-platform** (varies across the four downloads):
```
bin/
  clang              # (clang.exe on Windows)
  clang++
  lld
  blyt          # (blyt.exe on Windows)
```

**Shared RV32IMAFC target artifacts** (identical in all four downloads):
```
include/
  blyt.h
  blyt32.h
  blyt/
    audio.h, state.h, gfx.h, input.h, stage.h, ...

lib/
  libblyt32.so       # cart-facing variant library
  libblyt32lua.so    # Lua bindings
  libblytc.so        # runtime-provided C library (ADR-0120)
  libc++.a           # modified LLVM libc++ for C++ carts (see below)
  libc++abi.a

cmake/
  BlytToolchain.cmake
  BlytConfig.cmake

linker/
  cart.ld
```

`blyt_runtime.h` is absent from the cart SDK. It is the frontend embedding
API, not a cart author concern.

Target artifacts are built once (on a Linux x86-64 CI agent) and packaged
unmodified into all four downloads. Per-host binaries are built on their
respective platform agents (or cross-compiled where Rust/Cargo supports it).

### BlytConfig.cmake

`BlytConfig.cmake` is generated by CMake's install machinery and enables
`find_package(Blyt)` for cart authors who drive compilation with CMake
directly rather than `blyt`. It wires include paths, library paths,
toolchain configuration, and for C++ carts the static libc++ linker flags
(equivalent to `-static-libstdc++` but pointing at the SDK's modified
`libc++.a`). C++ cart authors do not discover these flags manually.

### Modified libc++

This amends ADR-0121: the SDK ships `libc++.a` and `libc++abi.a` built
from a **modified fork** of LLVM's libc++, not the stock upstream library.

The fork lives at `third_party/libcxx/` in the `blyt` repository (git
submodule pointing to the Blyt organisation's libc++ fork), following the
fork-as-submodule pattern established in ADR-0126.

**Modifications — removed headers and facilities:**

| Removed | Reason |
|---|---|
| `<filesystem>` | Cart data access is through the resource API (ADR-0040), not the filesystem. Absent from the SDK; including it is a compile error. |
| `<fstream>`, `<ifstream>`, `<ofstream>` | Filesystem-based stream I/O. `<sstream>` and `<iostream>` (for stdout formatting) remain. |
| `<thread>`, `<mutex>`, `<condition_variable>`, `<future>` | Carts are single-threaded; threading primitives have no valid implementation in the cart execution environment. |
| `<locale>` | Locale state is global mutable state that cannot be safely serialised into save state. Locale-dependent formatting is unavailable. |

**Modifications — stubs:**

| Stubbed | Behaviour |
|---|---|
| `std::random_device` | Construction calls `std::terminate()`. Cart code is compiled with `-fno-exceptions` (ADR-0121); this produces a clean crash at the point of construction rather than a cryptic seccomp `SIGSYS`. Cart authors should use the console RNG API (ADR-0041). |

**What remains:** the full algorithmic and container standard library —
`<algorithm>`, `<vector>`, `<map>`, `<unordered_map>`, `<optional>`,
`<variant>`, `<expected>`, `<memory>`, `<string>`, `<string_view>`,
`<span>`, `<sstream>`, `<numeric>`, `<functional>`, `<tuple>`, `<array>`,
`<bitset>`, `<chrono>` (duration arithmetic; no system clock), and the
rest of the non-I/O, non-threading standard library.

**Build.** The modified libc++ is cross-compiled for RV32IMAFC ILP32
using the bundled Clang, with `-fno-exceptions -fno-rtti` (matching
ADR-0121's mandatory cart compilation flags). Building libc++ from source
is slow; the `libc++.a` and `libc++abi.a` artifacts are cached in CI and
rebuilt only when the `third_party/libcxx` submodule pointer changes.

## Consequences

- Cart authors download the SDK for their platform, unzip, add `bin/` to
  PATH. No separate toolchain installation step; no version skew; no
  platform-specific configuration guides for the compiler.
- Windows is a first-class case. The bundled Clang and blyt cover
  the setup path that is hardest to document away.
- The modified libc++ makes the restricted cart execution environment
  visible at the header level: attempting to include `<fstream>` or
  `<thread>` is a compile error, not a runtime surprise or a cryptic
  seccomp signal.
- `std::random_device` stubbing produces a `std::terminate()` rather than
  a SIGSYS. A clear crash at the point of misuse is preferable to a signal
  that implicates the wrong location.
- The LLVM prebuilt model pins the SDK to a specific LLVM release. Toolchain
  updates require a new SDK release; they are infrequent in practice.
- The two-tier structure minimises CI build matrix: target artifacts are
  built once on a single agent; per-host binaries each require their native
  or cross-compiled build.
- Emscripten remaining external means WASM development has a separate setup
  step. This is acceptable; WASM targeting is an advanced scenario and
  emsdk's own tooling is self-contained.
- The universal macOS binary covers both Apple Silicon and Intel Macs with
  a single download. No per-architecture selection is required of users.
