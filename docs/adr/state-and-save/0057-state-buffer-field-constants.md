# ADR-0057: State buffer field access via fc_field_h compile-time integer constants

## Status
Accepted

## Context

State buffer field access needs to be both ergonomic for authors and fast at
runtime. String-based field lookups ("x", "y", "hp") are readable but require
a hash or linear search at every access. Numeric offsets are fast but opaque
and fragile under schema changes. The packer-generated constant pattern
(established in ADR-0009 and ADR-0040) resolves this: names in source, opaque
integer constants at compile time.

## Decision

**State buffer fields are accessed via `fc_field_h` compile-time integer
constants generated by the packer.**

`fc_field_h` is a `uint32_t` typedef. The upper bits encode the buffer ID;
the lower bits encode the field index within that buffer. This allows the
runtime to validate that a field constant belongs to the buffer it is being
used with, rather than silently reading the wrong field.

```c
typedef uint32_t fc_field_h;
#define FC_FIELD_NONE ((fc_field_h)0)  // invalid sentinel
```

Packer-generated constants:

```c
// cart_state.h (generated, gitignored)
#define S_PLAYER_X   ((fc_field_h)0x00010001)
#define S_PLAYER_Y   ((fc_field_h)0x00010002)
#define S_PLAYER_HP  ((fc_field_h)0x00010003)
```

Usage:

```c
fc_buffer_set_f32(S_PLAYERS, slot, S_PLAYER_X, new_x);
float y = fc_buffer_get_f32(S_PLAYERS, slot, S_PLAYER_Y);
```

In dev builds, the runtime validates that the field handle's embedded buffer
ID matches the `fc_buffer_h` argument. In release builds this check is
elided.

In Lua, the SOA metatable sugar (ADR-0011) wraps these constants so authors
write `players.x[i]` instead of using integer handles directly.

## Consequences

- Field access is a compile-time integer — no string lookup, no hash table.
- The embedded buffer ID enables dev-build validation of handle/buffer
  consistency, catching "wrong buffer" bugs that would be silent with raw
  offsets.
- `FC_FIELD_NONE` (zero) is an unambiguous invalid sentinel that the runtime
  can detect and report.
- Packer-generated files are gitignored; the packer regenerates them on every
  build, so schema changes are immediately reflected in constants.
