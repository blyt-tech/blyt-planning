# ADR-0081: Text mode — 80×30 character grid at 640×480 (deferred, v1.5+)

## Status
Deferred

## Context

A text mode enables roguelikes, text adventures, BBS-style interfaces, and
terminal aesthetics — genres and visual styles that are natural fits for a
fantasy console but awkward to implement using the graphical framebuffer API.

Classic 80-column text mode (as found in DOS VGA text mode) uses an 80×25
or 80×30 character grid with a 8×16 font at 640×400 or 640×480 output
resolution. This resolution was historically standard: VGA text mode was
*higher* resolution than many VGA graphical modes, because readable 80-column
text at 8×16 requires those pixels.

**Why 640×480 is appropriate here despite the console's 320×240 identity:**

The console's 320×240 framebuffer is always upscaled by the frontend to the
actual display resolution. Modern displays — including the lowest tier targets
(Chromebook, Pi 4 with 1080p output) — have far more pixels than 320×240.
640×480 is a different source resolution fed through the same frontend scaling
pipeline; it is not a break from the display architecture, just a different
declared framebuffer size.

A text mode within 320×240 would be severely constrained: an 8×16 font gives
only 40 columns (half the classic width); a 4×8 font gives 80×30 but is
marginal quality. Neither is the authentic 80-column DOS text experience.
640×480 at 8×16 is the right resolution for the right aesthetic.

## Proposed design

**Output:** 640×480 framebuffer. Text-mode carts declare this in the manifest;
the runtime and libretro adapter output at this resolution rather than 320×240.
Graphical carts are unaffected.

**Character grid:** 80 columns × 30 rows (80 × 8 = 640px, 30 × 16 = 480px).

**Character set:** CP437 (the IBM PC character set) — 256 characters including
the full box-drawing set, block elements, mathematical symbols, and the classic
smiley faces. Essential for roguelikes and BBS aesthetics.

**Character cells:** each cell encodes a character index plus a foreground and
background colour, each as an index into the console's 256-colour palette.
4 bytes per cell; full grid is 9,600 bytes.

The 256-colour palette model matches modern terminal colour support
(xterm-256color): 256 independently addressable colours per cell for both
foreground and background. This is more expressive than classic 16-colour
DOS text mode and consistent with how contemporary terminal applications
and roguelikes use colour. The palette itself is cart-defined (as in graphical
mode), so a cart can lay out its 256 colours however it likes — including
replicating the standard xterm-256color layout (16 named colours + 6×6×6
colour cube + 24-step greyscale ramp) if compatibility with existing terminal
colour schemes is desired.

**Font:** adaptive based on available output pixels:
- **Default:** a quality bitmap font of the Cozzette class — designed for
  pixel-accurate terminal rendering, clear at 8×16, with good CP437 coverage.
  Used when the output resolution supports it.
- **Fallback:** a minimal 4×8 font for severely constrained displays. Trades
  aesthetics for legibility at small sizes. The same 80×30 grid applies; the
  output framebuffer would be 320×240 in this mode.

The font adaptation is transparent to cart code: the text API addresses
character cells by column and row regardless of the underlying font.

**Cart API (sketch):**
```lua
blytty.text.put(col, row, char, fg, bg)
blytty.text.print(col, row, str, fg, bg)
blytty.text.fill(col, row, w, h, char, fg, bg)
blytty.text.scroll(top_row, bottom_row, lines)
blytty.text.cursor(col, row, visible)
blytty.text.clear(fg, bg)
```

Character constants for CP437 special characters (box-drawing, blocks, etc.)
generated by the packer or provided as a standard library.

**Manifest declaration:**
```yaml
# cart.info.yaml
display_mode: text   # default: graphics (320×240)
```

The runtime outputs 640×480 when `display_mode: text` is declared; the
libretro adapter reports the changed geometry to the frontend.

**Porting goal: NetHack.** NetHack is an explicit validation target for
text mode. It exercises the full character grid, CP437 box-drawing and
symbol characters, 256-colour terminal colour schemes, and complex
keyboard input. A working NetHack port demonstrates that the text mode
API is complete and that the CP437 character set and colour model are
correct.

**Bones files.** NetHack persists "bones files" between runs: when a
character dies on a given dungeon level, a bones file is written
recording the level state and dead character. Future runs may encounter
that level with the former character as a ghost or zombie. Bones files
are external files on the host filesystem, outside the normal save game
model — one of the first real porting obstacles beyond the display and
input layers.

Two options when porting:

- **Disable bones** (simplest): NetHack has a compile-time `BONES` flag.
  Disabling it produces a fully functional game without cross-run
  persistence. Appropriate for a first port that prioritises getting
  the game running.
- **Map to the cart save API**: bones files are named by dungeon level
  and character class (`bones.dlvl5.val`, etc.) and are modest in size.
  They can be stored as named slots via `blyt32.save.write` /
  `blyt32.save.read`, giving the full bones experience within the
  console's save infrastructure. The save quota (default 10 MB per
  cart) is ample for accumulating bones files over many runs.

The decision can be deferred to porting time; the save API already
provides the mechanism if bones support is desired.

## Open questions

- Should text-mode and graphical-mode output coexist within one cart (e.g.,
  a graphical game with a text-mode debug console)? Or are they strictly
  separate output modes?
- Cursor blink: hardware-style automatic blink driven by the runtime, or
  cart-controlled via the API?
- Input (see below).

### Input challenge

Text-mode carts have fundamentally different input requirements from
graphical carts. NetHack alone uses vi-keys for 8-directional movement
(hjklyubn), single-key commands across most of the keyboard, shift/ctrl
modifiers, and extended command prefixes. This does not map to the
console's 10-button abstract input model.

**Keyboard-primary model.** Text mode may only be fully functional when a
physical keyboard is present. The runtime detects keyboard availability
and can surface a warning or fallback UI when a text-mode cart is launched
without one. This is a reasonable constraint — text-mode carts are
naturally desktop/laptop experiences, and requiring a keyboard is honest
about what they need.

**Gamepad fallback.** A limited gamepad scheme is possible for simple
text-mode carts (D-pad for cursor/movement, face buttons for common
actions, shoulder+face combos for modifiers). Not sufficient for NetHack
but workable for purpose-built text-mode games that design for it
deliberately.

**On-screen keyboard.** A runtime-provided virtual keyboard (triggered by
a chord or Start) could handle the occasional command in a primarily
controller-driven text mode cart. Not suitable as the primary input
method for keyboard-heavy games.

**Proposed resolution:** text-mode carts declare their input requirements
in the manifest. A keyboard-required declaration causes the runtime to
warn (or refuse to start) when no keyboard is present, rather than
silently providing a degraded experience.

**Resolution: BlyTTY, the keyboard-primary sibling variant.** Rather than
extending Blyt32 with an awkward keyboard-required mode, the sibling
variant **BlyTTY** is the planned home for text mode (see ADR-0105 for the
variant model). BlyTTY shares the runtime infrastructure (RV32IMAFC ELF
cart format, Lua, save state, audio, lifecycle, `blytbuild` toolchain) but
declares a keyboard-primary input contract. Text mode at 640×480 is its
natural display mode rather than an optional extension. Carts for BlyTTY
target roguelikes, text adventures, interactive fiction, and BBS-style
tools — genres that are ill-served by a gamepad model.

BlyTTY is the practical forcing function for drawing the clean
variant/runtime boundary that §17 of the design document recommends
practising. It shares all the hard infrastructure; the differences are
input model, display resolution, and the default font and character set
bundled in the runtime binary. Carts link `libblytty.so` (and
`libblyttylua.so` for Lua carts) instead of the Blyt32 equivalents; the
Lua API surface is exposed under `blytty.*` instead of `blyt32.*`.

**Startup chime.** BlyTTY's boot audio identity should evoke an IBM XT or
PC/AT cold start: the single short POST beep from the PC speaker,
followed by the sounds of a hard disk spinning up and seeking. This is
immediately recognisable to the audience that grew up with DOS machines
and establishes the aesthetic before a single character appears on
screen.

## Why deferred

V1 establishes the graphical 320×240 identity. Text mode is a meaningful
addition — not a correction — and warrants dedicated design and
implementation attention rather than being squeezed into v1 scope.
The manifest `display_mode` field is a minor-version API addition (ADR-0032)
and does not require a major bump.

## Consequences (when implemented)

- Roguelikes, text adventures, BBS tools, and terminal emulators become
  natural cart types.
- The libretro adapter must handle variable output geometry (already
  supported by the libretro protocol).
- The CP437 font asset is bundled in the runtime binary alongside the
  existing default assets (ADR-0042).
- Graphical carts are entirely unaffected.
